# Manual Override Rules

## Override posture

Manual overrides must be exceptional, time-limited, auditable, reversible, and narrower than the original control they override. A manual override must not become an informal permanent enablement path.

## Required override fields

A future manual override record should include:

- reason for override;
- operator requesting override;
- override owner approving override;
- affected surface and environment;
- exact control being overridden;
- start time and expiration time;
- expected user or operator impact;
- cost, provider, model, billing, and evidence-boundary implications;
- audit location;
- rollback owner and rollback procedure;
- follow-up review requirement.

## Override limits

Manual overrides must not:

- silently enable product surfaces;
- bypass audit requirements;
- broaden audience beyond the approved scope;
- run models, providers, benchmarks, billing, or evidence promotion outside authorization;
- erase stop conditions;
- convert docs-only evidence into product-readiness evidence.

## Expiration and review

Every override should expire automatically or require a documented renewal. After expiration or rollback, the operator should record whether any stop condition, audit gap, user impact, cost event, provider call, model run, or evidence-boundary issue occurred.

## Adoption authority

Level 6 controls whether and how manual override rules are adopted.
