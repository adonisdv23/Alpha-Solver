# Candidate Task Schema Fields

## Scope

These are candidate task fields only. They are documentation vocabulary for future operator-only local execution and audit review. They are not a production database schema, migration, ORM model, API contract, dashboard contract, queue payload, or runtime implementation.

## Candidate task identity fields

| Field | Candidate meaning | Notes |
| --- | --- | --- |
| `task_id` | Stable local identifier for one operator-visible task. | Candidate string; may be generated by an operator script in a future lane. |
| `task_slug` | Human-readable task key. | Candidate kebab-case or snake_case alias. |
| `task_title` | Short operator-facing task title. | Candidate plain text. |
| `task_version` | Version of the task definition. | Candidate semantic or date-based version. |
| `task_created_at_local` | Local timestamp when the task record is drafted. | Candidate timestamp; no clock source is implemented here. |
| `task_created_by_operator` | Local operator label or pseudonymous handle. | Candidate audit field; avoid secrets or personal data unless explicitly required. |

## Candidate task instruction fields

| Field | Candidate meaning | Notes |
| --- | --- | --- |
| `task_objective` | Operator-readable objective. | Should be concise and bounded. |
| `task_allowed_changes` | Allowed file paths or document surfaces. | Docs-only lanes should list docs-only paths. |
| `task_disallowed_changes` | Explicit forbidden changes. | Should include runtime, provider, dashboard, API, database, queue, evidence promotion, and backlog workbook boundaries when applicable. |
| `task_required_files` | Required output files or artifacts. | Candidate list of repo-relative paths or artifact reference objects. |
| `task_required_content` | Required content elements. | Candidate checklist. |
| `task_decision_requirements` | Required selected next action, fallback lane, or decision markers. | Candidate checklist. |
| `task_checks_required` | Required checks before closure. | Candidate command list. |
| `task_return_requirements` | Required final response fields. | Candidate checklist. |

## Candidate task governance fields

| Field | Candidate meaning | Notes |
| --- | --- | --- |
| `task_evidence_boundary_label` | Boundary label for what the task can and cannot claim. | Example: `DOCS_ONLY_SCHEMA_DESIGN`. |
| `task_risk_class` | Operator-assigned risk class. | Candidate values: `docs_only`, `local_only`, `runtime_touch`, `provider_touch`, `production_touch`. |
| `task_requires_operator_confirmation` | Whether explicit operator confirmation is required before execution. | Candidate boolean. |
| `task_confirmation_profile` | Which confirmation fields apply. | See `operator-confirmation-fields.md`. |
| `task_stop_policy` | Stop policy for blocked, unsafe, or out-of-scope execution. | See `stop-reason-fields.md`. |
| `task_audit_notes` | Free-form audit note. | Should remain concise and evidence-bound. |
