# Risk Register

## Scope

This register identifies Self Operator failure modes before implementation. It records risks, impact, mitigations to consider in a future implementation lane, stop conditions, and blocked actions.

| ID | Risk | Failure mode | Impact | Mitigation requirement before implementation | Stop condition |
| --- | --- | --- | --- | --- | --- |
| SO-RR-001 | Polluted PR branches | Self Operator mixes unrelated edits, generated artifacts, backlog workbook changes, or preserved source artifacts into one branch. | Review becomes unreliable and unsafe changes can be hidden. | Require narrow write scopes, pre-commit changed-file review, and explicit changed-file allowlists. | Stop if changed files exceed the authorized lane or include sensitive/preserved paths. |
| SO-RR-002 | Stale PR updates | Self Operator updates a PR based on outdated branch state, outdated review comments, or stale selected-next docs. | Reviewers may approve obsolete evidence or miss conflicts. | Require branch freshness checks, clear rebase/merge policy, and stale-review detection. | Stop if base branch or decision docs changed after the operator formed its plan. |
| SO-RR-003 | Overclaimed evidence | Self Operator treats docs, smoke checks, or local outputs as production, benchmark, provider, MVP, or quality evidence. | Unsupported readiness claims can escape into PRs, docs, dashboards, or release notes. | Require evidence classification, blocked-claim checks, and reviewer approval for promotion. | Stop if a claim exceeds the recorded evidence boundary. |
| SO-RR-004 | Vague source evidence | Self Operator cites broad folders or generic memories instead of exact files, commands, and artifacts. | Reviewers cannot reproduce or audit decisions. | Require precise source-evidence logs with file paths and exact commands. | Stop if evidence cannot be traced to reviewed files or captured outputs. |
| SO-RR-005 | Implicit provider calls | Self Operator runs commands that call hosted providers, local models, fallback paths, or billable services without explicit approval. | Data exposure, cost, nondeterminism, and policy violation risk. | Require provider-call allowlists, dry-run defaults, environment checks, and fail-closed behavior. | Stop if a command could call a provider or fallback path outside approved scope. |
| SO-RR-006 | Accidental credential exposure | Self Operator reads, logs, commits, screenshots, or echoes secrets, tokens, private URLs, or environment contents. | Credential compromise and incident response burden. | Require secret redaction, no secret reads unless explicitly approved, and pre-commit secret scanning. | Stop if credentials or private URLs are present in output, diffs, docs, screenshots, or artifacts. |
| SO-RR-007 | Auto-merge risk | Self Operator merges, enables auto-merge, resolves review gates, or changes branch protections without human authorization. | Unsafe code/docs can enter protected branches without review. | Require explicit human merge authority and prohibit branch-protection changes. | Stop if a step requests merge, auto-merge, force-push, release, or branch-protection changes. |
| SO-RR-008 | Dashboard/API exposure | Self Operator exposes dashboards, routes, previews, health endpoints, metrics, or API surfaces while preparing evidence. | Unauthorized access, data leakage, billing abuse, or unsupported product claims. | Require separate product-surface security review before exposure. | Stop if dashboard/API route exposure is needed to complete the lane. |
| SO-RR-009 | Unsupported readiness claims | Self Operator claims Self Operator readiness, production readiness, provider readiness, API readiness, dashboard readiness, quality readiness, or release readiness from this packet. | Misleading status and unsafe downstream planning. | Require explicit readiness ladders and acceptance decisions before claims. | Stop if this packet is used as readiness evidence beyond risk-register existence. |
| SO-RR-010 | Unsafe action expansion | Self Operator infers permission to modify runtime, credentials, providers, CI, deployment, routing, SAFE-OUT, replay, observability, budgets, or SolverEnvelope behavior. | Sensitive behavior changes can bypass specs and tests. | Require explicit implementation contracts and scoped approvals for each sensitive area. | Stop if implementation is attempted under this docs-only packet. |
| SO-RR-011 | Evidence promotion drift | Self Operator moves raw artifacts, summarized artifacts, or local observations into promoted ledgers without review. | Artifact provenance and decision ledgers become unreliable. | Require promotion gate, provenance checklist, and reviewer sign-off. | Stop if evidence promotion is requested before boundary review. |
| SO-RR-012 | Review bypass by automation | Self Operator treats its own checks as sufficient review for high-risk actions. | Human oversight and separation of duties are lost. | Require independent reviewer approval for high-impact decisions. | Stop if the same actor proposes, executes, verifies, and approves a high-risk action. |

## Overall severity

The aggregate risk is high until future implementation work proves fail-closed controls for branch hygiene, provider boundaries, credential handling, evidence classification, route/dashboard exposure, and human approval gates.
