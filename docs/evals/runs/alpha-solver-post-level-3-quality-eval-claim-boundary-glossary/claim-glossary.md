# Claim Glossary

## Supporting-reference rule

This glossary is a supporting reference only and does not authorize any claim. A future packet must supply its own approved scope, execution evidence, artifacts, scoring method, and interpretation boundary before making any bounded claim.

## Claim categories

| Category | Meaning | Required boundary |
| --- | --- | --- |
| Quality claim | A statement that Alpha Solver improves, validates, measures, or proves answer quality. | Requires executed quality evaluation evidence and must state sample, rubric, scorer, and limits. |
| Benchmark claim | A statement that Alpha Solver has benchmark evidence, benchmark scores, benchmark wins, or benchmark comparability. | Requires benchmark execution under a documented protocol. |
| Alpha superiority claim | A statement that Alpha is better than plain prompting, competitors, other solvers, humans, or providers. | Requires comparative methodology, matched tasks, baselines, metrics, and uncertainty limits. |
| Product-readiness claim | A statement that the product is ready for users, release, launch, customer use, or operational use. | Requires a release-readiness packet and associated gates. |
| MVP readiness claim | A statement that Alpha Solver is MVP-ready or satisfies MVP acceptance. | Requires the specific MVP readiness evidence and acceptance criteria. |
| Production readiness claim | A statement that Alpha Solver is production-ready or safe for production traffic. | Requires production gates for reliability, security, operations, monitoring, incident response, and support. |
| Provider readiness claim | A statement that hosted or local providers are ready, validated, available, reliable, or supported. | Requires provider-specific execution evidence and operational boundaries. |
| Dashboard readiness claim | A statement that dashboards are ready, exposed, validated, complete, or production-capable. | Requires dashboard route, auth, UI, telemetry, and access validation. |
| `/v1/solve` readiness claim | A statement that `/v1/solve` is exposed, ready, stable, supported, production-capable, or validated. | Requires API execution evidence and deployment/access scope. |
| Billing readiness claim | A statement that billing, metering, pricing, quotas, payments, or cost controls are ready. | Requires billing-specific implementation and validation evidence. |
| Evidence-promotion claim | A statement that local evidence can be used in marketing, release notes, public claims, sales claims, or external benchmark claims. | Requires explicit evidence-promotion authorization. |

## Evidence-promotion boundary

Evidence promotion means using internal or local-only artifacts to support broader claims beyond their original scope. This packet blocks evidence promotion. Level 2 and Level 3 artifacts remain local-only, non-promotional references unless a later approved lane explicitly changes that boundary with adequate evidence.
