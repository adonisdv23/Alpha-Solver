# Claim-Boundary Map

| Display area | Allowed bounded statement | Blocked claim | Why blocked | Safe wording |
|---|---|---|---|---|
| artifact completeness | The committed source map marks required artifacts as present, missing, unknown, or future-required. | Complete artifacts prove quality/value/readiness. | Artifact presence is not output evaluation. | Artifact status only; no quality conclusion. |
| comparison state | The selected packet has or lacks Alpha/routed and plain/baseline side artifacts. | Alpha is better, superior, or benchmark-proven. | Comparison status is not scoring or interpretation. | Comparison artifacts are available or unavailable. |
| route/expert context | Route/expert/SAFE-OUT/confidence metadata is present, missing, or out of scope. | Route metadata proves correctness or value. | Metadata is diagnostic context only. | Route context can inform review but does not prove quality. |
| scoring state | Scoring is blank, blocked, reviewed, locked, or historical for a packet. | Scores create a broad value/readiness claim. | Scoring remains packet-bounded and may not be interpreted. | Scoring status is packet-bounded. |
| score-lock state | A lock artifact exists or is unknown. | Locked scores authorize unblinding or final interpretation. | Locking prevents mutation only. | Score lock does not authorize interpretation. |
| source identity state | Source identities are hidden, reviewed, unknown, or out of scope. | Source identities can be revealed now. | B015 authorizes no reveal. | Source identity remains hidden unless a committed reviewed packet already exists. |
| interpretation state | Final interpretation is blocked, missing, historical, or reviewed for a specific packet. | B015 performs final interpretation. | B015 is planning only. | Interpretation status only; no new interpretation. |
| check results | Claim-safety and diff checks passed/failed/not run. | Passing checks proves readiness. | Checks validate docs boundaries, not product capability. | Checks passed for this docs packet. |
| next action | One next safe operator action is review or static mockup planning. | Next action authorizes implementation/runtime/provider/scoring. | Separate operator approval is required. | Recommended next lane is static-only B016. |
